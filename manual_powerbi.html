Manual de Desarrollo de Power BI: Visualización de Datos

Esta guía recoge un flujo base para construir un dashboard de Power BI usando datos genéricos de negocio, con foco en buenas prácticas de desarrollo y mantenimiento.

Enfoque de Desarrollo

El objetivo no es solo "hacer gráficas", sino construir un informe mantenible: datos limpios, modelo coherente, métricas claras y visualización orientada a decisiones de negocio.

Alcance de esta versión: se usan ejemplos visuales genéricos creados para la guía (sin capturas de Windows), de forma que puedas reutilizar el proceso en cualquier proyecto.

1. Resumen de Fases de Desarrollo y Pasos Realizados

El desarrollo se ha organizado en cuatro fases. En cada una incluyo qué se hizo, por qué se hizo y qué validación se aplicó para asegurar calidad del dato.

Fase 1: Obtención de Datos

Carga inicial

Partimos de tres fuentes típicas: ventas, clientes y catálogo de productos. La extracción mantiene granularidad por fecha y por entidad para soportar análisis de detalle y agregados mensuales.

Fase 2: Transformación de Datos (Power Query)

Limpieza y normalización

En Power Query se aplican reglas de limpieza: tipado consistente, eliminación de duplicados, tratamiento de nulos y estandarización de categorías. El objetivo es dejar una capa de datos fiable antes del modelado.

Fase 3: Modelado de Datos

Modelo analítico

Se construye un esquema estrella con una tabla de hechos y dimensiones de fecha, cliente y producto. Se revisa cardinalidad 1:N y dirección de filtro para asegurar resultados correctos en DAX.

Fase 4: Visualización de Datos

Dashboard final

Se diseña una página de control con KPIs ejecutivos, tendencia temporal y análisis por categoría. El filtrado cruzado permite pasar de la visión global al detalle con un único contexto de filtros.

Validación funcional: antes de publicar, contrasta totales de ventas, recuento de clientes y margen con la fuente original para verificar que no haya desviaciones por modelado.

Regla 1: métrica única

Cada KPI tiene una única definición DAX para evitar cálculos distintos en páginas diferentes.

Regla 2: modelo ligero

Se eliminaron columnas no analíticas en origen para reducir peso del dataset.

Regla 3: trazabilidad

Las transformaciones críticas se documentaron para facilitar mantenimiento.

2. Explicación Técnica de las Herramientas

Power Query

Power Query es el bloque ETL del proyecto. Aquí se definieron los pasos de estandarización (tipos, columnas, valores faltantes, uniones y reglas de limpieza). La ventaja es que el proceso queda automatizado y se replica en cada actualización sin intervención manual.

DAX (Data Analysis Expressions)

DAX aporta la capa de lógica analítica. Se empleó para medidas dinámicas de negocio, por ejemplo: coste por hora efectiva, facturación por periodo, saldo de bolsa de horas y porcentaje de desviación. Todo cálculo se diseñó respetando contexto de filtro para soportar segmentaciones por fecha y proyecto.

Ejemplo de medidas DAX del proyecto
Horas Totales = SUM(TrabajoDiario[horas])

Facturacion Total = SUM(Facturacion[importe_final])

Desviacion Horas % =
DIVIDE(
    [Horas Totales] - SUM(Proyectos[horas_estimadas]),
    SUM(Proyectos[horas_estimadas]),
    0
)

Gráficas (Visualizaciones)

Las visualizaciones se organizaron por objetivo: diagnóstico rápido (tarjetas KPI), análisis temporal (línea/área), composición de costes (barras apiladas) y control operativo (tablas de detalle). El filtrado cruzado entre visuales permite pasar de visión ejecutiva a detalle técnico en pocos clics.

KPI priorizados en esta guía:
  • Horas consumidas vs horas contratadas.
  • Facturación mensual acumulada.
  • Margen operativo estimado por proyecto.
  • Proyectos con riesgo por sobreconsumo.

3. Guía de Desarrollo Reutilizable

Checklist que he seguido para poder rehacer el dashboard desde cero en cualquier iteración del proyecto:

  • Definir tabla de hechos principal y granularidad temporal antes de crear visuales.
  • Normalizar IDs y claves de relación desde Power Query para evitar errores en DAX.
  • Crear primero medidas base (horas, coste, ingreso), luego métricas derivadas.
  • Aplicar formato de página y tema al final, no al inicio, para no perder tiempo en iteraciones.
  • Comparar siempre los totales de Power BI con el ERP antes de publicar informe.

Este enfoque deja el manual preparado para evolución: nuevos campos, nuevas métricas y nuevas páginas de informe, manteniendo coherencia técnica y visual.
